# E2Eシナリオ — kouji-nou-support-app

**【Round2見直し 2026-07-14】** 訓練提案のパーソナライズ廃止に伴い、E2E-02, E2E-03, E2E-05, E2E-06を改訂した。

## E2E-01: 正常系1 — 本人が予定を登録し通知を受け取り実施する標準フロー

1. **前提条件**: DRY_RUN=true、テスト用本人アカウントでログイン済み、PWAとしてインストール済み（FR-SYS-001）
2. **入力データ**: 予定「9:00 服薬」（type=服薬、datetime=当日9:00）
3. **実行手順**: FR-NTF-001で予定登録 → PushScheduler(CW-01)が起動時刻を検出（DRY_RUNのためログ出力のみ）→ アプリ画面で予定が一覧表示される
4. **期待結果**: 予定がホーム画面に表示され、[DRY_RUN] Push would be sent ログが出力される
5. **検証方法**: UI表示確認 + DRY_RUNログの内容確認（TC-NTF-001, TC-NTF-002）
6. **タイムアウト**: 30秒
7. **クリーンアップ**: テスト用予定データを削除

## E2E-02: 正常系2 — 訓練メニュー一覧を閲覧する最小フロー【Round2改訂】

1. **前提条件**: DRY_RUN=true、新規本人アカウントでログイン済み
2. **入力データ**: なし（訓練メニュー画面への遷移のみ）
3. **実行手順**: 訓練メニュー画面を開く（FR-TRN-001）
4. **期待結果**: 3カテゴリ全てのメニューが**全ユーザー共通の内容で**表示され、免責表示（FR-TRN-004）が画面上部に表示される
5. **検証方法**: UI表示確認、免責文言の完全一致確認（TC-TRN-001, TC-TRN-004）
6. **タイムアウト**: 15秒
7. **クリーンアップ**: 不要（データ変更なし）

## E2E-03: 正常系3 — 介護者が代理で予定・チェックリスト・訓練メニュー閲覧を一通り操作するフルフロー【Round2改訂】

1. **前提条件**: DRY_RUN=true、本人アカウントが招待コードを発行済み
2. **入力データ**: 招待コード、予定1件、チェックリスト1件
3. **実行手順**: 介護者が招待コード入力(FR-USR-002) → 予定登録(FR-ADM-001) → チェックリスト登録(FR-ADM-001) → 訓練メニュー画面を確認(FR-TRN-001)
4. **期待結果**: 全操作が成功し、本人側画面にも同一データが反映される。ProxyLogに全操作が記録される(FR-ADM-002)
5. **検証方法**: E2Eテストで各画面遷移・データ整合性・ProxyLogレコード数を確認（TC-USR-002, TC-ADM-001, TC-ADM-002, TC-TRN-001）
6. **タイムアウト**: 45秒
7. **クリーンアップ**: テスト用アカウント・関連データを全削除

## E2E-04: 異常系1（外部障害） — オフライン時にPush通知が届かずキャッチアップ表示で補償される

1. **前提条件**: DRY_RUN=true、予定が登録済み（datetime=過去、notified=false）、端末をオフラインに設定
2. **入力データ**: なし
3. **実行手順**: オフライン状態でアプリを起動する（FR-SYS-003, FR-NTF-003）
4. **期待結果**: Push通知は届かないが、「見逃した予定」欄に該当予定が表示される
5. **検証方法**: ネットワークをモックで遮断した状態でのE2Eテスト（Playwright `context.setOffline(true)`）、UI表示確認（TC-NTF-003, TC-SYS-003）
6. **タイムアウト**: 30秒
7. **クリーンアップ**: テスト用予定データを削除、オフライン設定を解除

## E2E-05: 異常系2（不正な入力/フロー） — 招待コードの総当たり攻撃対策確認【Round2改訂】

1. **前提条件**: DRY_RUN=true、本人アカウントが招待コード発行済み、テスト用介護者アカウントでログイン済み
2. **入力データ**: 誤った招待コードを5回連続入力
3. **実行手順**: 誤った招待コードを5回入力する（FR-USR-002の例外条件を確認）
4. **期待結果**: 5回目の失敗後、以後1時間は同一本人アカウントへの招待コード入力が拒否される（Firestore試行カウンタによるブロック）
5. **検証方法**: E2Eテストで6回目の入力（正しいコードでも）が拒否されることを確認、Security Rulesレベルでの拒否をAPIレベルでも確認（TC-USR-002）
6. **タイムアウト**: 20秒
7. **クリーンアップ**: テスト用試行カウンタ・アカウントを削除

## E2E-06: 境界値 — 招待コードの有効期限境界（24時間）【Round2改訂】

1. **前提条件**: DRY_RUN=true、本人アカウントが招待コードを発行済み（発行時刻を操作可能なテスト環境）
2. **入力データ**: 発行から23時間59分経過した招待コード／発行から24時間1分経過した招待コード
3. **実行手順**: 23時間59分時点で入力（境界: 有効）→ 24時間1分時点で入力（境界: 無効）
4. **期待結果**: 23時間59分時点では紐付け成功。24時間1分時点では「期限切れ」エラーとなり紐付けされない（FR-USR-002の境界条件）
5. **検証方法**: 単体テストで招待コード検証関数を直接呼び出し、発行時刻からの経過時間の境界値（24時間ちょうど前後）で結果を比較（TC-USR-002）
6. **タイムアウト**: 10秒
7. **クリーンアップ**: テスト用招待コードデータを削除
