## 1. はじめに

### 1.1 目的
高次脳機能障害者（本人）とその介護者が、生活上の記憶・注意・遂行機能の困難を補い、公的ガイドラインに基づく訓練情報を無理なく得られるスマートフォン向けPWAを提供する。

### 1.2 範囲
`research/scope_table.md` のIN SCOPE機能（予定・服薬リマインド、日課チェックリスト、介護者代理編集、訓練メニュー一覧表示【Round2: 全ユーザー共通・パーソナライズなし】）を対象とする。医療機関連携・AI自由生成訓練・障害プロフィールに基づくパーソナライズ・法人向け機能はOUT OF SCOPE。

### 1.3 用語集

| 用語 | 定義 |
|---|---|
| 本人 | 高次脳機能障害の当事者、アプリの主利用者 |
| 介護者 | 本人の生活を支える家族・支援者、代理操作を行う利用者 |
| SaMD | Software as a Medical Device（プログラム医療機器）。薬機法上の規制対象となりうるソフトウェア |
| 要配慮個人情報 | 個人情報保護法第2条3項に定める、障害の事実等を含む特別な配慮を要する個人情報 |
| PWA | Progressive Web App。ホーム画面に追加してネイティブアプリのように使えるWebアプリ |

### 1.4 参考文書
- `research/intake_sheet.md`、`research/scope_table.md`
- `research/research_v1.md`、`research/research_v2.md`
- 厚労省「プログラムの医療機器該当性に関するガイドライン」（令和5年一部改正）

## 2. システム概要

本アプリは PWA として提供され、本人と介護者が同一アプリ内でロールに応じた機能を利用する。オフラインファースト設計（IndexedDB）とクラウド同期（Firestore）を併用し、コストゼロインフラ（**【Round3改訂】GitHub Pages/Actions** + Firebase Sparkプラン）上で稼働する。

### コンテキスト図（概略）

```
[本人スマホ] ──PWA──┐
                    ├──HTTPS──> [GitHub Pages: 静的配信]
[介護者スマホ] ─PWA─┘                    │
                                          ▼
                    [GitHub Actions(5分間隔): NotificationCheckJob]
                                          │
                                    (VAPID Push)
                                          ▼
                          [ブラウザPush Service (FCM/APNs経由)]

[本人/介護者スマホ] ──HTTPS/SDK──> [Firebase Firestore(Spark) + Authentication]
```

### 主要機能の概略
1. リマインド通知（予定・服薬）
2. 日課チェックリスト
3. 介護者代理登録・編集
4. 訓練メニュー一覧表示（公知情報ベース、【Round2】全ユーザー共通・パーソナライズなし・専門家への相談を促す免責表示付き）

## 7. 機能要件

### 7.1 ユーザー管理（USR）

#### FR-USR-001: ロール選択によるアカウント作成
- **優先度**: Must
- **Given**: ユーザーが初回起動している
- **When**: ユーザーが「本人」または「介護者」を選択してアカウントを作成する
- **Then**: 選択したロールに応じた初期画面が表示される。ロール情報がFirestoreに保存される
- **例外**: ロール未選択のまま次へ進もうとした場合、選択を促すメッセージを表示し進行をブロックする

#### FR-USR-002: 介護者の代理アクセス権限取得
- **優先度**: Must
- **Given**: 本人が招待コード（**8桁英数字**、【Round1修正 CX-06】要配慮個人情報へのアクセス権を付与するため6桁から強化）を発行済みである
- **When**: 介護者が招待コードを入力する
- **Then**: 介護者アカウントが本人アカウントに紐付けられ、代理操作権限（FR-ADM-001）が付与される
- **例外**: 招待コードが不正または**期限切れ（発行から24時間、【Round1修正 CX-06】72時間から短縮し攻撃対象期間を縮小）**の場合、エラーメッセージを表示し紐付けを行わない。**同一本人アカウントへの入力試行が5回失敗した場合、Firestore上の試行カウンタ（Security Rulesで検証）により以後1時間の入力を拒否する（総当たり攻撃対策）**

### 7.2 認証（AUTH）

#### FR-AUTH-001: メール/パスワード認証
- **優先度**: Must
- **Given**: ユーザーがメールアドレスとパスワード（8文字以上）を入力している
- **When**: ユーザーがログインを実行する
- **Then**: Firebase Authenticationによる認証が成功し、ロールに応じたホーム画面へ遷移する
- **例外**: 認証失敗時は「メールアドレスまたはパスワードが正しくありません」を表示する。**【Round1修正 CX-05】連続失敗時のロックはFirebase Authentication標準の指数バックオフ機構（サーバー側で自動的に再試行間隔を延伸）に準拠する。SparkプランではCloud Functionsを使わない前提のため、独自の「60秒ロック」等のカスタムサーバー側ロック実装は行わない**

### 7.3 障害プロフィール・同意（DATA）— 【Round2で廃止】

**【Round2見直し 2026-07-14】** 訓練提案機能のパーソナライズ廃止に伴い、旧FR-DATA-001（要配慮個人情報の個別同意取得）およびFR-DATA-002（障害プロフィール入力）は**廃止**した。本アプリは障害の種類・重症度等の要配慮個人情報を一切収集しない設計に変更したため、これらの要件・同意フロー自体が不要になった（黄金ルールの禁止事項「IDの意味を変えて番号を使い回すこと」に従い、FR-DATA-001/002は欠番として扱い、再利用しない）。

| 廃止ID | 廃止理由 |
|---|---|
| FR-DATA-001 | 要配慮個人情報を収集しないため同意フロー自体が不要 |
| FR-DATA-002 | 障害プロフィールの収集自体を廃止（訓練メニューは全ユーザー共通表示に変更、FR-TRN-001参照） |

### 7.4 リマインド通知（NTF）

#### FR-NTF-001: 予定・服薬登録
- **優先度**: Must
- **Given**: ユーザーがログイン済みである
- **When**: 本人または介護者が予定（日時・タイトル・種別[一般/服薬]）を登録する
- **Then**: 予定がローカルとFirestoreに保存され、一覧に表示される
- **例外**: 日時が過去日時の場合、警告を表示するが登録は許可する（介護者が記録目的で使う場合を考慮）

#### FR-NTF-002: Push通知送信
- **優先度**: Must
- **Given**: 予定が登録済みで、対象端末がPWAとしてホーム画面に追加され通知許可済みである
- **When**: 予定の指定時刻に到達し、端末がネットワークに接続している
- **Then**: GitHub Actions(NotificationCheckJob)経由でVAPID Push通知が送信され、端末に通知が表示される（配信目標: 指定時刻から**15分以内**。【Round3改訂】GitHub Actionsのscheduleトリガーは最短5分間隔かつ高負荷時は遅延しうるため、Cloudflare Workers採用時の「60秒以内」から現実的な値に修正）
- **例外**: 端末がオフラインの場合、通知は配信されない（FR-NTF-004でキャッチアップ表示により補償する。NFRとして許容する技術的制約とする）

#### FR-NTF-003: キャッチアップ表示
- **優先度**: Must
- **Given**: 通知配信時刻を過ぎた未実施の予定が存在する
- **When**: ユーザーがアプリを開く
- **Then**: ホーム画面上部に「見逃した予定」として一覧表示する
- **例外**: 7日以上前の見逃し予定は「古い予定」として折りたたみ表示する

#### FR-NTF-004: 音声読み上げ通知（オプション）
- **優先度**: Should
- **Given**: ユーザーが設定で音声読み上げを有効にしている
- **When**: 通知が表示される
- **Then**: Web Speech API（またはブラウザ標準TTS）で通知内容を読み上げる
- **例外**: 対応ブラウザでない場合、読み上げをスキップし通常通知のみ表示する

### 7.5 日課チェックリスト（DATA）

#### FR-DATA-003: チェックリスト項目登録
- **優先度**: Must
- **Given**: ユーザーがログイン済みである
- **When**: 本人または介護者がチェックリスト項目（タイトル・頻度[毎日/曜日指定]）を登録する
- **Then**: 項目がローカルとFirestoreに保存される
- **例外**: タイトル未入力では登録不可

#### FR-DATA-004: チェックリスト完了記録
- **優先度**: Must
- **Given**: 本日分のチェックリスト項目が表示されている
- **When**: 本人が項目をタップする
- **Then**: 完了状態（完了/未完了）と完了日時がローカルとFirestoreに保存され、タップ領域は44x44px以上のボタンで表示される
- **例外**: 誤タップ対策として、再タップで完了を取り消せる（Undo）

### 7.6 介護者代理編集（ADM）

#### FR-ADM-001: 代理登録・編集
- **優先度**: Must
- **Given**: 介護者がFR-USR-002により本人アカウントに紐付けられている
- **When**: 介護者が予定・チェックリスト・障害プロフィールを登録または編集する
- **Then**: 変更が本人アカウントのデータとして保存され、本人側画面にも反映される
- **例外**: 紐付けが解除されている場合、編集操作はサーバー側で拒否される

#### FR-ADM-002: 代理操作ログ記録
- **優先度**: Should
- **Given**: 介護者が代理操作を実行した
- **When**: 操作が完了する
- **Then**: 操作者ID・日時・操作内容（変更前後の差分）がログとして90日間保持される
- **例外**: ログ容量がFirestore無料枠を圧迫しないよう、90日超過分は自動削除する

### 7.7 訓練メニュー提案（TRN）— 【Round2で簡略化: パーソナライズ廃止】

**【Round2見直し 2026-07-14】** 開発時間とSaMD（医療機器プログラム）該当性リスクを最小化するため、障害プロフィールに基づく絞り込み・パーソナライズを廃止した。訓練メニューは**全ユーザーに同一内容を表示する静的な一般情報**とする。これにより、厚労省ガイドラインのフローチャート③C「利用者への情報提供のみ」（research_v2.md参照）に、個人差なくより明確に該当する設計となり、法的リスクがさらに低下する。

#### FR-TRN-001: 公知情報ベースの訓練メニュー一覧表示（全ユーザー共通）
- **優先度**: Must
- **Given**: 本人または介護者が訓練メニュー画面を開く
- **Then**: 記憶訓練/注意訓練/遂行機能訓練の3カテゴリ全てが、公的ガイドライン（国立障害者リハビリテーションセンター公開資料等）に基づく静的な一覧として、**個人差なく全ユーザーに同一内容で**表示される。各メニューには参照元（一次情報URL）を表示する
- **例外**: なし（ログイン後は常に同一の一覧を表示する）

#### 廃止: FR-TRN-002（プロフィールに基づく絞り込み）、FR-TRN-003（実施結果に基づく難易度自動調整）
パーソナライズ機能の廃止に伴い、両要件は欠番とする（番号の再利用はしない）。

#### FR-TRN-004: 免責表示
- **優先度**: Must
- **Given**: 訓練メニュー画面が表示される
- **When**: 画面が読み込まれる
- **Then**: 「本アプリは医療機器ではなく、疾病の診断・治療・予防を目的としていません。訓練は医療従事者の指導の代替ではなく補助を目的としています。**【Round2追記】無理に行わず、体調や気分に応じて休んでください。実施にあたって不安な点や、ご自身に合っているか分からない場合は、必ず主治医・作業療法士等の専門家にご相談ください**」という免責文を画面上部に常時表示する
- **例外**: なし（常時表示、非表示設定は提供しない）

#### FR-TRN-006: AIによる自由生成の訓練コンテンツ（対象外）
- **優先度**: Won't
- **内容**: 訓練メニューをAIが動的・自由に生成する機能は今回実装しない（scope_table.md OUT OF SCOPE参照）
- **除外理由**: SaMD該当性リスク（ADR-006）、誤った訓練提案による安全性リスクを回避するため。訓練メニューは`src/data/trainingMenus.json`による静的・公的ガイドライン準拠の管理のみとする

#### FR-TRN-005: 疾病候補・重症度判定の非表示（SaMD非該当維持）
- **優先度**: Must
- **Given**: 訓練メニュー画面（【Round2】障害プロフィール機能自体が廃止されたため対象は訓練メニュー画面のみ）
- **When**: システムがデータを処理・表示する
- **Then**: 疾病名・疾病候補・医学的重症度スコアを一切算出・表示しない。表示するのは公的ガイドラインに基づく静的な訓練メニュー一覧（全ユーザー共通）のみとする
- **例外**: なし（実装レベルでの機能制限。将来のいかなる機能追加時もこのMust要件を優先する）

### 7.8 システム（SYS）

#### FR-SYS-001: PWAインストール対応
- **優先度**: Must
- **Given**: ユーザーが対応ブラウザ（iOS Safari 16.4+/Android Chrome最新2バージョン）でアプリにアクセスしている
- **When**: Web App Manifestが読み込まれる
- **Then**: 「ホーム画面に追加」操作でPWAとしてインストールできる
- **例外**: 非対応ブラウザの場合、Web版としての利用は可能だが通知機能が使えない旨を表示する

#### FR-SYS-002: インストール誘導オンボーディング
- **優先度**: Must
- **Given**: ユーザーが初回起動し、かつPWAとしてインストールされていない
- **When**: 初期画面が表示される
- **Then**: 「通知機能を使うにはホーム画面への追加が必要です」という説明画面と操作手順（OS別）を表示する
- **例外**: 既にPWAとしてインストール済みの場合、この画面はスキップする

#### FR-SYS-003: オフライン時のローカル機能継続
- **優先度**: Must
- **Given**: 端末がオフライン状態である
- **When**: ユーザーが予定・チェックリスト・訓練メニューの閲覧または操作を行う
- **Then**: IndexedDBに保存されたローカルデータを用いて機能が継続する（Push通知を除く）
- **例外**: オンライン専用機能（介護者招待コード発行等）はオフライン時に利用不可であることを表示する

#### FR-SYS-004: オンライン復帰時の自動同期
- **優先度**: Should
- **Given**: オフライン中にローカルで変更が行われた
- **When**: 端末がオンラインに復帰する
- **Then**: ローカルの変更差分をFirestoreに自動同期する
- **例外**: 同期競合（同一データが介護者・本人双方で変更された場合）は最終更新時刻が新しい方を優先する

## 9. データ要件

### 9.1 主要エンティティ

| エンティティ | 主なフィールド | 保存先 |
|---|---|---|
| User | userId, role(本人/介護者), linkedUserId, email | Firestore + Firebase Auth |
| Schedule | scheduleId, userId, datetime, title, type(一般/服薬), completed | IndexedDB + Firestore |
| ChecklistItem | itemId, userId, title, frequency, completedLogs[] | IndexedDB + Firestore |
| ~~DisabilityProfile~~ | 【Round2廃止】要配慮個人情報を収集しない方針のため本エンティティ自体を削除 | - |
| ~~TrainingRecord~~ | 【Round2廃止】難易度自動調整の廃止に伴い実施記録の保存も不要化。訓練メニューは`src/data/trainingMenus.json`の静的データのみで完結 | - |
| ProxyLog | logId, actorUserId, targetUserId, action, diff, timestamp | Firestore |

### 9.2 データライフサイクル
- ローカル(IndexedDB): 端末保存、アンインストールまで保持
- クラウド(Firestore): アカウント削除時に全データ削除（GDPR的な削除権に準拠する運用）
- ProxyLog: 90日で自動削除（FR-ADM-002）

### 9.3 バックアップ
無料インフラのためバックアップSLAは提供しない。Firestoreの標準的な耐久性に依拠する（NFR-REL-001に明記）。

## 11. セキュリティ・監査

- **認証**: Firebase Authentication（メール/パスワード）、Blazeプラン誘導機能は不使用（NFR-SEC-001）
- **認可**: 本人は自分のデータのみアクセス可。介護者はFR-USR-002で紐付けられた本人データのみアクセス可（Firestore Security Rulesで実装）
- **秘密情報管理**: VAPID鍵・Firebase設定は環境変数、リポジトリにコミットしない
- **監査ログ**: FR-ADM-002の代理操作ログを90日保持
- **【Round2修正】要配慮個人情報を一切収集しない設計となったため、以下のインシデント対応フローは主にメールアドレス等の一般個人情報漏洩を想定した予防的措置として維持する（リスクレベルはRound1時点より大幅に低下）**
- **インシデント対応（【Round1修正 OPUS-005】具体化）**: 個人情報の漏洩（不正アクセス・誤送信・設定ミス等）を検知した場合、以下の手順で対応する
  1. 検知後直ちに該当機能を停止（GitHub Pagesのロールバック(revert push)またはGitHub Actionsワークフローの無効化）
  2. 漏洩範囲（対象ユーザー数・データ種別）を特定
  3. **速やかに**（個人情報保護法上は「速やかに」と規定、目安として3〜5日以内）個人情報保護委員会に報告（該当様式は個人情報保護委員会公式サイトから入手）
  4. 影響を受ける本人（またはその介護者）に対し、漏洩内容・対応状況を通知
  5. PROGRESS.mdに経緯・対応・再発防止策を記録
  - 本フローは非エンジニアの社長でも実行できるよう、チェックリスト形式で`harness/incident-response-checklist.md`に別途詳細化する（Phase6着手時に作成）
