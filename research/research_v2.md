## SaMD（プログラム医療機器）該当性判定 ★重要

厚労省「プログラムの医療機器該当性に関するガイドライン」（令和3年制定/令和5年改正、別紙フローチャート）を精読した結果、以下の設計指針が確定した。

### フローチャート適用結果

1. **フローチャート②**: 本アプリは「医療関係者が業務に使用しない（個人使用のみ）」に該当 → フローチャート③へ進む
2. **フローチャート③A（データの表示・保管・転送のみか）**: 訓練提案機能があるため単純なYESにはならない
3. **フローチャート③B（生活環境改善等の医療・健康以外の目的のみか）**: **要注意** — 脚注に「リハビリテーション...を意図するものは、疾病の診断、治療、予防に使用することが目的であると考えられるため①CへYESに進む」と明記されている。**訓練提案機能を「リハビリテーション」と標榜すると医療機器該当性判断（フローチャート①C以降）に進んでしまう**
4. **フローチャート③C（利用者への情報提供のみを行うものか）**: **これがYESとなる設計にすることが安全策**。ガイドライン中のYES例に「健康な人の健康維持・増進のために、一般的な食事、運動、体重管理等の生活習慣に関するアドバイスを行うもの」「統計情報の提供」が明記されている
5. **フローチャート③D（入力情報を基に疾病候補を表示するか）**: 障害プロフィール入力から「疾病候補」を提示する設計にしないことが必須（NOを維持）

### 確定した設計指針（Must要件として反映）

| # | 指針 | 根拠 |
|---|---|---|
| SaMD-1 | 訓練提案機能は「リハビリテーション」「治療」という語を標榜せず、「生活サポート情報の提供」「参考メニューの提示」と位置づける | フローチャート③B脚注 |
| SaMD-2 | 訓練メニューは独自アルゴリズムによる重症度判定を行わず、公的ガイドライン（公知の情報）に基づく参考情報の提示に留める | フローチャート③C YES条件、公知情報の定義 |
| SaMD-3 | 入力された障害プロフィールから「疾病候補」「重症度」を判定・表示する機能は実装しない | フローチャート③D |
| SaMD-4 | UI・広告文言に「診断」「治療」「改善を保証」等の医療行為を想起させる表現を使用しない | 8(1)標ぼうの留意事項 |
| SaMD-5 | 免責表示として「本アプリは医療機器ではなく、疾病の診断・治療・予防を目的としていません」を明記する | 8(1)標ぼうの留意事項（望ましい対応として記載） |
| SaMD-6 | 難易度自動調整はチェックリストの実施結果（○/×等の単純な達成度）に基づく機械的な処理に留め、認知機能の医学的評価を行わない | ③C/③D境界の安全マージン確保 |

## 技術スタック（確定）

**【Round3改訂 2026-07-14】ホスティング・Push通知送信の行を更新。旧内容（Cloudflare Pages/Workers）は末尾の「Round3追記」を参照。**

| レイヤー | 技術 | バージョン/プラン | 選定理由 |
|---|---|---|---|
| フロントエンド | React + Vite | Vite 5+ | 高速ビルド、エコシステム充実、Q5希望に合致 |
| PWA化 | vite-plugin-pwa (Workbox) | 最新安定版 | ゼロコンフィグでService Worker生成、活発にメンテ（2026-05更新） |
| UIコンポーネント | Radix UI or Chakra UI | 最新安定版 | WAI-ARIA準拠、アクセシビリティ要件（Q6）に直結 |
| ローカルDB（オフライン） | Dexie.js（IndexedDBラッパー） | v4系 | オフラインファースト、Star 14.5k、活発にメンテ |
| ホスティング | **GitHub Pages** | 無料プラン | 社長要望によりGitHubへ統一、非商用利用のため規約上問題なし、帯域月100GB(ソフト上限)で十分 |
| Push通知送信 | **GitHub Actions（schedule、5分間隔）+ web-push** | 無料プラン | GitHubへ統一。1分間隔だったCloudflare Workersより粗いが実用上許容（詳細はRound3追記） |
| データ永続化（クラウド同期） | Firebase Firestore | **Sparkプラン固定**（Blazeへ切替禁止） | 無料枠内（読取5万/書込2万/日）で数百ユーザー規模をカバー、超過時ブロックのみ |
| 認証 | Firebase Authentication | 無料枠 | メール/パスワード認証は無料。Blaze誘導される機能（電話番号認証等）は不採用 |

**運用上の必須ルール**: Firebaseコンソールで請求先アカウントを紐付けず、Blazeへの誤アップグレードを防止する。

## 【Round3追記 2026-07-14】インフラをCloudflareからGitHubへ変更

社長要望により、ホスティング・通知基盤をCloudflare(Pages/Workers)からGitHub(Pages/Actions)へ変更した。

### 変更の一次情報確認結果

| # | 調査項目 | 結果 | ソースURL | 信頼度 |
|---|---|---|---|---|
| 1 | GitHub Pagesの商用利用制限 | 「online businessとしての利用」は禁止されているが、本アプリは非商用の個人向け無償アプリのため該当しない。帯域は月100GB（ソフト上限） | https://docs.github.com/en/pages/getting-started-with-github-pages/github-pages-limits | 高(A) |
| 2 | GitHub Actions scheduleトリガーの最短間隔 | **最短5分間隔**（公式に明記）。高負荷時は遅延の可能性あり | https://docs.github.com/en/actions/using-workflows/events-that-trigger-workflows#schedule | 高(A) |
| 3 | GitHub Actions scheduleの自動無効化条件 | パブリックリポジトリで**60日間活動がないと自動的に無効化**される | 同上 | 高(A) |

### 旧決定との差分・影響

- Cloudflare Workers採用時は1分間隔だったが、GitHub Actionsでは5分間隔が上限。NFR-PERF-002（Push配信目標）を「60秒以内」→「15分以内」に修正（`docs/SRS.md`参照）
- 60日間無活動での自動無効化リスクに対し、月次の運用チェック（`PROGRESS.md`運用責任者欄）でリポジトリ活動を維持する運用ルールを追加（`harness/HARNESS.md` H6）
- Cloudflare Workers向けに設計していたWebCrypto自前JWT実装（旧ADR-009）は、GitHub Actionsが実Node.js環境であるため不要になり、公式`firebase-admin` SDKに置き換え、設計がむしろ単純化された（`docs/SDD.md` ADR-010）
- 制約: iOSはPWAホーム画面追加必須、Web Pushはネットワーク接続必須（完全なオフライン通知は不可能という技術的限界を受容）
